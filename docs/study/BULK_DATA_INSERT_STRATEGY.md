# 대용량 데이터 적재 방법 정리

## 1. 대용량 데이터 적재가 필요한 이유

검색 기능의 성능을 확인하려면 적은 데이터만으로는 한계가 있다.

데이터가 수십 건 또는 수백 건 수준일 때는 검색 조건, 정렬, 페이징, 캐시 적용 여부에 따른 성능 차이가 잘 드러나지 않는다.
따라서 실제 서비스와 유사한 환경을 만들기 위해 50만 건 이상의 상품 데이터를 적재하고, 검색 API의 응답 속도를 비교할 필요가 있다.

이번 작업에서는 상품 검색 성능 테스트를 위해 대량의 더미 상품 데이터를 생성하고 적재하는 방법을 검토했다.

---

## 2. 대용량 적재 방식 후보

### 2-1. JPA save 반복

가장 단순한 방법은 JPA Repository의 `save()`를 반복 호출하는 방식이다.

```java
for (int i = 0; i < 500_000; i++) {
    productRepository.save(product);
}
```

이 방식은 구현이 쉽고, 엔티티 기반으로 데이터를 다룰 수 있다는 장점이 있다.

하지만 대용량 데이터 적재에는 적합하지 않다.
JPA는 영속성 컨텍스트를 사용하기 때문에 저장한 엔티티들이 메모리에 계속 쌓일 수 있다.
그 결과 데이터가 많아질수록 메모리 사용량이 증가하고, `OutOfMemoryError`가 발생할 가능성이 있다.

이를 방지하려면 일정 개수마다 `flush()`와 `clear()`를 호출해야 한다.

```java
if (i % 1000 == 0) {
    entityManager.flush();
    entityManager.clear();
}
```

즉, JPA 방식은 사용할 수는 있지만 대량 적재만을 목적으로 할 경우 성능과 메모리 측면에서 부담이 있다.

---

### 2-2. JDBC Batch Insert

두 번째 방법은 `JdbcTemplate.batchUpdate()`를 사용하는 방식이다.

JDBC Batch Insert는 여러 건의 INSERT SQL을 일정 단위로 묶어서 DB에 전달한다.

```text
1,000건 생성
→ batchUpdate 실행
→ 리스트 비우기
→ 다음 1,000건 생성
```

이 방식은 JPA 영속성 컨텍스트를 거치지 않기 때문에 대용량 적재에 더 적합하다.

또한 전체 데이터를 한 번에 메모리에 올리는 것이 아니라, 1,000건 또는 5,000건 단위로 나누어 적재하면 메모리 사용량도 안정적으로 관리할 수 있다.

이번 작업에서는 상품 데이터를 빠르게 적재하는 것이 목적이므로, JPA save 반복보다는 JDBC Batch Insert 방식이 더 적합하다고 판단했다.

---

### 2-3. Datafaker

Datafaker는 더미 데이터를 생성해주는 라이브러리다.

상품명, 문장, 주소, 회사명, 숫자 등 다양한 랜덤 데이터를 만들 수 있다.

```java
Faker faker = new Faker(new Locale("ko"));
String description = faker.lorem().sentence();
```

다만 Datafaker는 데이터를 “넣는 방법”이 아니라 “데이터를 만드는 방법”이다.

즉, JDBC Batch Insert와 경쟁하는 기술이 아니다.

```text
JDBC Batch Insert = 데이터를 빠르게 넣는 방법
Datafaker = 넣을 데이터를 랜덤하게 만드는 방법
```

따라서 실제 사용 시에는 Datafaker로 더미 데이터를 만들고, JDBC Batch Insert로 DB에 적재하는 조합을 사용할 수 있다.

---

## 3. 최종 선택 방향

이번 대용량 검색 테스트에서는 다음 방식이 적합하다고 판단했다.

```text
Datafaker + 직접 키워드 조합
+ JdbcTemplate.batchUpdate()
```

상품명은 검색 테스트를 위해 의도적으로 키워드를 포함하도록 생성한다.

예를 들어 다음과 같은 키워드를 사용한다.

```text
아이폰, 맥북, 갤럭시, 에어팟, 키보드, 마우스, 의자, 책상, 나이키, 가방
```

그리고 설명에는 Datafaker를 활용해 랜덤 문장을 섞을 수 있다.

이렇게 하면 검색 테스트에 필요한 키워드는 유지하면서도, 데이터가 너무 단순해지는 것을 방지할 수 있다.

---

## 4. batch 단위를 나누는 이유

대용량 데이터를 적재할 때 가장 주의해야 할 점은 전체 데이터를 한 번에 메모리에 올리지 않는 것이다.

예를 들어 50만 건의 데이터를 한 번에 리스트에 담은 뒤 INSERT하면, 데이터 생성 과정에서 메모리 사용량이 크게 증가할 수 있다.

따라서 일정 단위로 나누어 처리해야 한다.

```text
전체 500,000건
batch size 1,000건

1,000건 생성 후 INSERT
1,000건 생성 후 INSERT
1,000건 생성 후 INSERT
...
```

JPA 방식에서는 일정 단위마다 `flush()`와 `clear()`를 호출해 영속성 컨텍스트를 비워야 한다.

JDBC Batch 방식에서는 일정 단위마다 `batchUpdate()`를 실행하고, 사용한 리스트를 비우면 된다.

핵심은 동일하다.

```text
대량 데이터를 한 번에 메모리에 들고 있지 않는다.
```

---

## 5. 정리

대용량 데이터 적재 방식은 목적에 따라 선택해야 한다.

JPA `save()` 반복은 구현은 쉽지만, 영속성 컨텍스트에 엔티티가 누적될 수 있어 대량 적재에는 부담이 있다.

JDBC Batch Insert는 SQL을 직접 작성해야 하지만, JPA 영속성 컨텍스트를 거치지 않고 빠르게 데이터를 넣을 수 있어 대용량 적재에 적합하다.

Datafaker는 데이터를 적재하는 방식이 아니라, 테스트용 더미 데이터를 생성하는 도구다.

따라서 이번 검색 성능 테스트에서는 Datafaker로 실제와 유사한 데이터를 만들고, JdbcTemplate의 batchUpdate를 사용해 일정 단위로 나누어 적재하는 방식을 선택한다.
