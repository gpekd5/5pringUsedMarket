# PROJECT_CONTEXT.md

## 프로젝트 개요

본 프로젝트는 중고거래 커머스 플랫폼 백엔드 프로젝트다.

서비스 컨셉은 당근마켓과 유사한 중고거래 플랫폼이다.

구매 요청이나 결제 시스템은 두지 않고, 구매 의사는 판매자와 구매자 간 채팅으로 협의한다. 판매자는 자신의 상품 상태를 `ON_SALE`, `RESERVED`, `SOLD`로 변경한다.

## 핵심 기능

- 회원가입 / 로그인 / 로그아웃 / 토큰 재발급
- JWT 인증 / 인가
- Redis 기반 Refresh Token 관리
- Redis 기반 Access Token Blacklist 로그아웃
- 상품 등록 / 목록 / 상세 / 수정 / 삭제
- 판매 상태 변경 / 예약 취소
- 상품 검색 v1
- 상품 검색 v2 with Local Cache
- 인기검색어 / 최근검색어
- 관심상품 등록 / 취소 / 목록 조회
- 선착순 쿠폰 발급
- Redis Lock 기반 동시성 제어
- 거래 채팅
- CS 문의 채팅
- WebSocket + STOMP
- 채팅 메시지 저장 및 커서 기반 조회
- 마이페이지 요약 조회
- Docker / GitHub Actions / EC2 배포

## 역할 분담

- 김현승: 인증/인가, 공통, 마이페이지, 배포
- 상품 담당: 상품 등록/조회/수정/삭제, 상품 상태 관리
- 쿠폰/동시성 담당: 선착순 쿠폰, Redis Lock, 동시성 테스트
- 검색/캐싱 담당: 상품 검색, 인기검색어, Local Cache
- 채팅 담당: 거래 채팅, CS 채팅, WebSocket/STOMP

## 핵심 기술 포인트

### 인증/인가

- Access Token 30분 + Refresh Token 14일 구조
- Refresh Token은 Redis 저장
- Refresh Token은 Redis Whitelist 방식으로 관리
- 토큰 재발급 시 Refresh Token Rotation 적용
- 재발급 성공 시 Redis Refresh Token을 새 값으로 교체
- 로그아웃 시 Access Token을 Redis Blacklist에 등록
- Blacklist TTL은 Access Token 남은 만료 시간
- Refresh Token 삭제로 재발급 차단

### 동시성 제어

- 선착순 쿠폰 발급에서 동시성 문제 발생 가능
- Redis Lock으로 쿠폰별 발급 로직 보호
- 이벤트 기간, 잔여 수량, 중복 발급 여부를 검증
- 동시성 테스트 코드로 해결 전/후 검증

### 캐싱

- 상품 검색 v1은 캐시 없음
- 상품 검색 v2는 Caffeine Local Cache 적용
- 인기검색어는 Redis ZSet 활용 가능
- README에 캐시 적용 이유와 성능 비교 기록

### 실시간 채팅

- WebSocket + STOMP 사용
- STOMP CONNECT 시점에 JWT 인증
- 채팅 메시지는 DB 저장
- 메시지 조회는 커서 기반 페이징
- 다중 서버 확장 시 Redis Pub/Sub 사용 가능

### 배포

- Docker 기반 컨테이너화
- GitHub Actions CI/CD
- AWS EC2 배포
- MySQL / Redis는 Docker Compose 또는 운영 환경 분리

## 설계 문서 구분

- ERD: MySQL 테이블 관계만 표현
- 애플리케이션 아키텍처: Spring Boot, MySQL, Redis 연결 표현
- 배포 아키텍처: GitHub Actions, Docker, EC2 표현
- Redis는 ERD가 아니라 아키텍처 문서에 표현

## 프로젝트 확정 정책

- 공통 구현 규칙과 문서 우선순위는 `AGENT_RULES.md`를 따른다.
- API 계약은 `API_SPEC.md`, 데이터 구조는 `ERD_GUIDE_FOR_CODEX.md`, 패키지와 책임은 `CODE_CONVENTION_USED_MARKET.md`를 기준으로 한다.
- 루트 패키지는 `com.example.fivespringusedmarket`을 사용한다.
- 상품 삭제 상태는 `DELETED`, 관심상품 URL은 `/wishes`를 사용한다.
- 검색 v2에는 Caffeine Local Cache를 적용한다.
- 발제 원문의 Redis Remote Cache는 도전 학습 항목이며 현재 프로젝트 구현 범위에서는 제외한다.
- Redis는 인증 토큰, Access Token Blacklist, 쿠폰 Lock, 인기검색어 ZSet, 채팅 Pub/Sub에만 사용한다.

### **3단계 : ERD 작성하기**

화면에 무엇이 있는지 살펴보기

- 우선 와이어프레임에서 각 화면에 어떤 정보들이 있는지 봐요. 예를 들어 로그인 화면이 있으면 `사용자` 정보가 필요하겠죠? 이렇게 화면에 등장하는 중요한 요소들을 찾는 거예요.

중요한 요소들을 entity로 만들기

- 방금 찾은 중요한 요소들을 entity로 바꿔볼까요? 각각의 entity는 데이터베이스에서 하나의 테이블로 생각할 수 있어요!

entity 안에 들어갈 정보 정하기

- 각 entity에 어떤 정보가 들어갈지 정해요. 예를 들어 `사용자` entity에는 이름, 이메일, 비밀번호 같은 게 필요하겠죠. 이건 와이어프레임에서 입력하거나 표시되는 항목들을 참고하면 돼요.

entity끼리의 관계 설정하기

- entity들간에 어떻게 연결될지 정해요. 관계를 잘 설정해줘야 데이터가 서로 잘 이어져요. 연관 관계를 설정할 때에는 entity 간의 적절한 `cardinality` 를 잘 설정해야해요.

사용자의 관점에서 데이터 흐름 살펴보기

- 이제 와이어프레임과 ERD를 비교하면서, 내가 설정한 데이터 구조가 실제 화면에서 잘 동작할지 확인해요. 혹시 놓친 부분이 없는지 동료들과 같이 살펴보는 것도 좋아요. 이렇게 하면 와이어프레임을 자연스럽게 데이터베이스 구조로 연결할 수 있어요. 생각보다 어렵지 않죠?

<aside>

**"검색 한 번에 DB까지 가야 해요?"**

---

인기 검색어와 상품 검색이 동시에 쏟아지는 커머스 플랫폼에서, 매번 DB를 조회하는 것은 비효율적입니다. 캐싱을 활용하면 자주 요청되는 데이터를 메모리에 저장해 응답 속도를 높이고 시스템 부하를 줄일 수 있어요. 로컬 캐시의 한계를 느끼고 Redis 같은 분산 캐시로 확장하는 과정까지 경험하면, 캐싱 전략을 설계하는 실무 역량을 기를 수 있습니다.

**📌 발표 시 반드시 짚어야 할 포인트**

1. **"왜 캐싱을 적용했나요?"** 
    - 검색 API에 캐시를 적용한 이유와 어떤 성능 병목을 해결하려 했는지
2. **"어떤 캐싱 전략을 선택했고, 왜?"** 
    - Cache-aside, Write-through 등 전략 선택 근거
3. **"캐시 Key는 어떻게 설계했나요?"** 
    - Key 충돌 방지, prefix 전략, TTL 설정 기준
4. **"로컬 캐시의 한계는 무엇이었나요?"**
    - Scale-out 환경에서의 문제점 (도전: Redis 전환 이유)
5. **"캐시 데이터와 원본 데이터가 다르면?"** 
    - 캐시 무효화(Eviction) 전략과 trade-off

**🔍 기대하는 발표 깊이**

> 단순히 "캐싱을 적용했더니 빨라졌다"가 아닌, **왜 그 전략을 선택했고, 어떤 한계를 인지하고 있는지**까지 설명할 수 있어야 합니다.
> 

**📊 발표 자료 준비**

- 캐시 적용 **전/후 성능 비교** 결과 (응답 시간, TPS 등 — 시각적 표, 그래프)
- 인기 검색어 랭킹이 캐시에서 어떻게 관리되는지 흐름도
- (도전) 캐시 없음 → 로컬 캐시 → Redis 캐시 **3단계 성능 비교**
- (도전) K6등 부하테스트 도구의 성능 테스트 보고서
    - TPS, 평균 응답 시간, 포화 지점(Saturation Point)

    - 낙관적 / 비관적 / 분산 락의 관리 주체, 보호 범위, 성능 특성 비교
