# [Troubleshooting] 상품 변경 시 검색 캐시가 갱신되지 않는 문제

## 1. 배경

검색 V2 API에서는 동일한 검색 조건의 반복 요청에 대해 DB 조회를 줄이기 위해 Caffeine Local Cache를 적용했다.

검색 요청이 들어오면 `productSearch` 캐시에 동일한 key가 있는지 확인하고, 캐시가 존재하면 DB를 조회하지 않고 캐시된 검색 결과를 반환한다.

```text
검색 요청
→ productSearch 캐시 확인
→ Cache Hit
→ DB 조회 생략
→ 캐시 결과 반환
```

이를 통해 동일 검색 조건 반복 요청에서는 DB 조회를 줄일 수 있다.

---

## 2. 문제 상황

검색 결과 캐시는 상품 데이터에 영향을 받는다.

예를 들어 다음과 같은 상품 변경이 발생할 수 있다.

```text
상품 등록
상품 수정
상품 삭제
상품 상태 변경
상품 이미지 변경
```

하지만 기존 캐시 구조에서는 검색 결과가 이미 캐시에 저장된 상태에서 상품이 변경되더라도, 캐시가 만료되기 전까지는 이전 검색 결과가 그대로 반환될 수 있었다.

예를 들어 사용자가 `맥북`을 검색해서 검색 결과가 캐시에 저장되었다고 가정한다.

```text
keyword=맥북
→ 맥북 상품 A
→ 맥북 상품 B
```

이후 판매자가 상품 A의 상태를 변경한다.

```text
상품 A 상태 변경
ON_SALE → SOLD
```

하지만 캐시가 아직 만료되지 않았다면 다음 검색 요청은 DB를 조회하지 않고 캐시된 결과를 반환한다.

```text
다시 keyword=맥북 검색
→ Cache Hit
→ DB 조회 생략
→ 기존 캐시 결과 반환
```

그 결과 판매완료 처리된 상품이 여전히 기존 상태로 검색 결과에 노출될 수 있다.

---

## 3. 원인 분석

원인은 검색 캐시에 TTL만 설정되어 있었고, 상품 변경 시점에 캐시를 직접 무효화하지 않았기 때문이다.

현재 Caffeine 캐시는 다음과 같은 설정을 가진다.

```text
TTL: 5분
maximumSize: 500
```

TTL이 있으면 일정 시간이 지나 캐시가 자동으로 만료된다.

하지만 TTL이 만료되기 전까지는 상품 변경 사항이 검색 결과에 즉시 반영되지 않을 수 있다.

즉, TTL은 오래된 캐시를 eventually 제거하는 역할은 하지만, 상품 변경 시점의 즉시 무효화를 보장하지는 않는다.

---

## 4. 해결 방법 검토

상품 변경 시 검색 캐시를 처리하는 방법은 크게 두 가지가 있다.

| 방법 | 설명 | 장점 | 단점 |
| --- | --- | --- | --- |
| TTL에만 의존 | 캐시 만료 시간까지 기다린다. | 구현이 단순하다. | 변경된 상품 정보가 일정 시간 늦게 반영된다. |
| 상품 변경 시 CacheEvict 적용 | 상품 변경 메서드에서 검색 캐시를 삭제한다. | 변경 이후 검색 결과가 최신 데이터로 갱신된다. | 변경 시 캐시가 비워지므로 일시적으로 캐시 효율이 떨어질 수 있다. |

검색 결과는 강한 정합성이 필요한 데이터는 아니지만, 상품 상태나 삭제 여부는 사용자에게 잘못 노출되면 혼란을 줄 수 있다.

따라서 TTL만 사용하는 것보다 상품 변경 시 검색 캐시를 무효화하는 방식이 더 적절하다고 판단했다.

---

## 5. 해결 방법: 상품 변경 시 CacheEvict 적용

상품 변경이 발생하는 메서드에 `@CacheEvict`를 적용했다.

```java
@CacheEvict(
        cacheNames = "productSearch",
        allEntries = true
)
```

`cacheNames`는 검색 V2에서 사용하는 캐시 이름이다.

```text
productSearch
```

`allEntries = true`는 해당 캐시 저장소의 모든 key를 삭제한다는 의미이다.

---

## 6. allEntries = true를 사용한 이유

검색 캐시는 검색 조건 조합별로 key가 생성된다.

```text
keyword=맥북:category=ALL:status=ON_SALE:sortType=LATEST:page=0:size=10
keyword=맥북:category=DIGITAL:status=ON_SALE:sortType=LATEST:page=0:size=10
keyword=ALL:category=DIGITAL:status=ON_SALE:sortType=PRICE_ASC:page=0:size=10
```

상품 하나가 어떤 검색 key에 포함되어 있는지 정확히 알기 어렵다.

예를 들어 상품 A가 다음 검색 결과에 모두 포함될 수 있다.

```text
keyword=맥북
category=DIGITAL
status=ON_SALE
sort=LATEST
page=0
```

상품이 수정되었을 때 특정 key 하나만 삭제하면 다른 검색 key에는 여전히 오래된 상품 정보가 남을 수 있다.

따라서 이번 프로젝트에서는 단순하고 안전한 방식으로 `productSearch` 캐시 전체를 무효화했다.

---

## 7. 적용 결과

상품 변경 시 검색 캐시가 삭제되도록 변경한 후 흐름은 다음과 같다.

```text
상품 수정/삭제/상태 변경
→ productSearch 캐시 전체 무효화
→ 다음 검색 요청
→ Cache Miss
→ DB 조회
→ 최신 검색 결과 캐시 저장
```

이를 통해 상품 변경 이후에도 검색 결과가 오래된 캐시를 계속 반환하는 문제를 줄일 수 있었다.

---

## 8. 고려한 한계

`allEntries = true`는 단순하고 안전하지만, 캐시 전체를 비우기 때문에 캐시 효율이 떨어질 수 있다.

예를 들어 상품 하나가 수정되었을 뿐인데 모든 검색 캐시가 삭제된다.

하지만 현재 프로젝트는 학습용 팀 프로젝트이고, 검색 캐시 key와 상품 데이터의 매핑 정보를 별도로 관리하지 않는다.

따라서 정밀한 key 단위 무효화보다는 전체 무효화를 선택했다.

추후 서비스 규모가 커지면 다음 개선을 검토할 수 있다.

```text
상품 ID 기반 캐시 역인덱스 관리
검색 조건별 캐시 key 추적
Redis 기반 캐시 무효화 전략
짧은 TTL + 이벤트 기반 캐시 삭제 조합
```

---

## 9. 정리

이번 문제는 검색 결과 캐시를 적용한 뒤 상품 데이터 변경 시 캐시 최신성을 어떻게 보장할 것인가에 대한 문제였다.

TTL만으로도 일정 시간이 지나면 오래된 캐시가 제거되지만, 상품 수정/삭제/상태 변경 직후에는 이전 검색 결과가 계속 노출될 수 있었다.

이를 해결하기 위해 상품 변경 메서드에 `@CacheEvict(cacheNames = "productSearch", allEntries = true)`를 적용하여 검색 캐시를 무효화했다.

이 과정을 통해 캐시는 조회 성능을 높이는 데 도움이 되지만, 데이터 변경 시점에는 반드시 캐시 무효화 전략을 함께 고려해야 한다는 점을 확인했다.
